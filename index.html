Collapsed

Collapsed is an open world multiplayer survival shooter in which you will scavenge a post-apocalyptic world looking for items to trade and build your home base with. While out in the world, you will encounter other players and fight for survival.

Studio:

Role:
Designer, Generalist
Engine:
Unity
Platforms:
PC

Things I worked on

As the team was very small, I have been involved in almost every aspect of the project. This includes doing gameplay programming, writing custom netcode, doing game design, implementing character animations and doing UI design.

A brief summary of features I worked on:

  • Custom netcode and interest management
  • Input and camera controller
  • Character movement and animation
  • Equipping clothing, weapons and attachments
  • Hit registration and ragdoll
  • Furniture carrying
  • Main menu and manual
  • World streaming using Addressables

Things I learned

Being my first professional project, the list of things I learned is endless. Looking back, some highlights include:

Custom netcode

Because at the time there were no multiplayer frameworks that met our requirements, we decided to build it ourselves. Although this was quite daunting at first, it gave us a lot of control over interest management and performance. Having a low-level understanding of netcode has also helped a lot in making multiplayer design decisions.

Integrating 3D assets

As we didn't have a 3D-artist on the team, we had to purchase these assets and customize them to our liking. With no prior knowledge, this turned out to be a long and bumpy ride. After spending more time than I'd like to admit creating spiderwebs in Mecanim, I learned how to retarget animations, use inverse kinematics and eventually ended up spending quite some time in Blender.

Performance

Building a large open world that performs with up to a hundred players has been challenging. As the game has realistic graphics, graphic performance has always been our biggest concern. After getting to know the profilers, I learned how to write performant code, improve graphic performance, find memory leaks and how to use Addressables to unload unused parts of the world.

Things I worked on

Being a smaller two man project, I have been involved in just about everything. This includes writing editor scripts, doing UI & UX design, writing documentation and providing customer support.

A brief summary of features I worked on:

  • Editor windows & preferences
  • Occlusion-data painting scene
  • Runtime components

Things I learned

Creating the tool and making it production-ready has taught me a couple of new things:

Unity Job System

In the first version of the tool, all calculations were done at runtime. This turned out to be too expensive to run on the main-thread, causing hiccups. By learning how to use the Unity Job System we were able to spread out the workload and run it in the background, solving the performance issue.

UI & UX design

Because our in-house tool was now going to be used by customers, we had to make it more intuitive and give it a better overall look. Looking at it from a customer perspective has helped a lot in getting rid of unnecessary configuration steps and has lead to new features to give users better feedback when something goes wrong.

Customer support

Providing customer support has been challenging yet rewarding. I lost count of how many times my skill in inductive reasoning has been put to the test after receiving a concise message or just a screenshot saying something is wrong. But at the end of the day, every bug report and suggestion has made the tool a little bit better and it has been great to hear stories from happy customers.

Things I worked on

Because this was another project with just the two of us, I have been involved in most things. This includes doing gameplay programming and game design.

A brief summary of features I worked on:

  • Networking player input
  • Character movement
  • Player spawning and teleporting
  • Obstacle course elements
  • Random obstacle course generation
  • Creating teams and win conditions

Things I learned

Although this was a relatively short project, I managed to learn a thing or two:

Advanced netcode

Using Photon's brand new networking framework Fusion has taught me how to put more advanced networking concepts into practise. With their guidance I was able to implement fully authoritative character movement with client-side prediction, something we had attempted before but never managed to get right. It was also really educative to see how they had implemented the more basic features and the reasoning behind their approach.

Architecture

A fresh start was a great opportunity to look critically at our previous codebase and make improvements. Coming from a heavily object-oriented architecture, we found out it doesn't work very well for programming games. After learning about a couple of design patterns that promote modularity, our codebase gained a lot of flexibility. This was a very welcome change when we were prototyping and things were changing rapidly.
